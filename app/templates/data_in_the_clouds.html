Data in the clouds





Summary

Most organisations are moving their solutions from being "on premise" to cloud-based solutions.

Why are organisations are investing in cloud computing?

The advantages of cloud computing are similar to "rent over buy" - you don't have the upfront capital expenditure of investing in your server infrastructure, and can account for ongoing expenses as operational expenditure offsetting profits in order to reduce corporation tax bill. Costs are usually a mixture of rental of part of the cloud server infrastructure (usually charged on a usage basis), service costs e.g. backup/restore and licensing costs (can be charged on a user, storage or usage basis). It's important to check that the service level in particular meets your requirements.

Is the Cloud secure?

Many companies remain skeptical about the information security aspects of cloud computing, but access to data is very secure with your company data seperated from other company's data and access over the network only available via very sophisticated encrypted public/private/session keys using a Secure Sockets Layer (SSL) (in addition to the usual authentication mechanisms). This is the same network encryption that is used to access your personal banking website, for example. So if you trust the security for that, you should trust the network security to your cloud provider. In Azure Security Center, you can also restrict access to the cloud to certain ip ranges and further restrict access to certain virtual machines.

AWS & Azure

This post provides a sample technology stack for the 2 most popular cloud systems - AWS and Azure - in order to provide a traditional data warehouse/reporting solution. The choice that you make will typically be a mixture of cost, your supplier's familiarity with the technology stack and user acceptance. Each organisation should make it's own choices. It's worth saying that neither AWS nor Azure natively provide much on the cloud, other than database engines, that meets the requirements, so you have to choose software provided by 3rd party suppliers or utilise on-premise software, where necessary.

Requirements

I'll present system architectures for a typical retail reporting requirement -

  • Create a data warehouse that stores customer (from prospect through sales, payment & fulfillment party roles), product (sold) and the associated sales & marketing campaign information (including information about the web events captured on the website (part of the digital platform) for a small/medium sized company.
  • Assume that volumes of web logs are not large, and that there is no need for real time analysis of those logs.

Azure information reporting system architecture

Microsoft's main enterprise cloud offering for developers is called Azure. Coupled with the user-friendly Office365 cloud (on which the various Microsoft Office solutions are available), this makes a compelling rival to AWS and has grown against it's rival in 2016.

Here, I show the web logs being captured from the web analytics tool of your choice e.g. Google or Adobe Analytics (as data source 1) on the left of the picture. Typically, these logs would be shipped on a periodic basis to a file storage layer (flow 1). You might use snaplogic (if the web logs are internally available) or sftp (if your website is hosted externally) in order to facilitate this movement. You could also stream the analytics logs to your solution.

As well as web logs, you will also need the relational data held in your sales & marketing database (which I've assumed for simplicity also holds your customer & product master data). Flow 2 shows SnapLogic (as an example) capturing relational data from these systems in near real time and depositing in the data staging layer. Flow 3 has SnapLogic parse the raw web logs and extract relevant fields from it that you wish to report on. For example, you may be interested in extracting information about the ip address that was used to browse your website and what products that ip address viewed on the website. You would then be able to tie that ip address to a customer, if they subsequently provide their details.

At this stage, you have raw data captured from your web logs and sales and marketing database available to you in the data staging layer of your data warehouse. You could simply use SQL to query the data at this stage, but as we're building a data warehouse to do proper integration and record history, we'll then need a data integration with sophisticated extract, transform & load capabilities. Fortunately, SnapLogic is able to act both as an enterprise service bus (ESB) and as an Extract, Transform & Load (ETL) tool. Note: Azure data services provides an Azure data factory as a very simple cloud-based ETL tool, but is unlikely to provide the capabilities that you will need to do full ETL. Another alternative is to use your existing on-premise ETL tool e.g. SSIS (a SQL*Server data integration service), if it is able to connect to Azure SQL Datawarehouse as a target. (It will be able to do this if it can handle jdbc, odbc or SQL*Server database targets already)

Once we've successfully integrated our data in to our data warehouse base layer, we'll then need to add in any business metrics and hierarchies of interest to our end customer (the posh name is a semantic layer). For example, sales may want us to calculate average sales order value and count. It's typically best to create these types of things as SQL views, as it allows you to re-create the metrics more easily than if you physically store the data. It's also better to create business metrics as SQL views than creating them within a reporting tool itself, because typically users will want to use more than 1 reporting tool, which means that if you'll have to re-create the metrics in each and every reporting tool if you don't create them in the data warehouse itself.

We then need to have some slick reports & dashboards. In the old days, companies tended to run scheduled reports which were then emailed to a group email account or even printed. Nowadays, a more modern BI approach is simply to have reports & dashboards available on the cloud, so that any user can execute the report from their desktop, tablet or phone. Power BI Pro (for which you will need an Office365 for business cloud account) is a good Microsoft choice for the modern BI cloud-based approach. For scheduled reports, you could go for Microsoft SSRS (part of SQL*Server 2016). More popular reporting & dashboard choices include Qlik, Tableau, Cognos, Microstrategy, OBIEE, SAP Business Objects.

Finally, in order that our user base understands the fields that appear in the report and dashboard we'll need a data dictionary or business glossary in order to store field definitions. For Azure, the Azure data catalog provides a cloud-based capability in this area.

AWS information reporting system architecture

AWS was the original developer-friendly cloud. It's most suitable for creating a digital platform (website and associated software) and hence heavily favoured by web application developers. AWS marketplace allows you to add on a variety of popular vendor software to your AWS cloud in order to provide you with an enterprise level offering.

I've picked some choices from AWS marketplace (shown in the diagram below) which would provide a similar solution to the Azure Cloud architecture presented above. Again, it is only the datawarehouse engine - Amazon Redshift - which is provided by Amazon itself. Compared to Azure, Redshift has some issues in that it is based on a relatively old version of PostgresQL, whereas Azure SQL Data Warehouse is actually more up to date than it's on-premise SQL*Server 2016 cousin (all be it lacking the useful associated services). I could not find an equivalent to the Azure Data Catalog on AWS, so have added Collibra, which is a much more fully functioned business glossary/data dictionary/data governance offering.
